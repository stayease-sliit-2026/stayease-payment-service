9.1 Introduction

My assigned component in the StayEase system is the Payment and Notification Service. This microservice is responsible for processing booking payments, issuing refunds, and recording/sending payment-related notifications (email/SMS). The service supports the overall booking lifecycle by ensuring that every confirmed booking can transition through a payment workflow with traceable transaction status updates.

This service plays a critical role in the full platform because:
- It is invoked by the Booking Service to process financial actions.
- It validates identity through the Auth Service before protected operations.
- It provides payment status APIs that other services and the frontend can query.
- It maintains notification history to support user communication and auditability.

The implementation is developed using Node.js (Express), MongoDB Atlas, Docker, GitHub Actions CI/CD, and SonarCloud for DevSecOps support.


9.2 Architecture & Design

The service is designed using a layered microservice architecture:
- Routes layer: API endpoint definitions and middleware chaining.
- Controllers layer: request/response orchestration.
- Services layer: business logic and external service integration.
- Models layer: MongoDB schema definitions.
- Middleware layer: authentication, internal authorization, validation, and error handling.

Design decisions and rationale:
1. Separation of concerns: each layer has a single responsibility, improving maintainability and testability.
2. Database-per-service principle: Payment and Notification data are isolated in this service domain.
3. Externalized configuration: runtime behavior controlled through environment variables for secure deployment.
4. Secure-by-default middleware pipeline: bearer token checks on protected routes and an internal key for internal-only notification operations.
5. Validation-first request handling: express-validator is applied before business logic execution.

Core entities:
- Payment: bookingId, userId, amount, currency, status (pending/success/failed/refunded), transactionId, timestamps.
- Notification: userId, bookingId, paymentId, channel, destination, message, status, timestamps.


9.3 API Endpoints

Implemented endpoints:
1. GET /health
- Purpose: Service health check.
- Auth: No.
- Response: service status and timestamp.

2. POST /payments
- Purpose: Process a payment for a booking.
- Auth: Bearer token required.
- Request fields: bookingId, userId, amount, currency, optional contact object.
- Response: created payment object and status.

3. GET /payments/:bookingId
- Purpose: Retrieve payment status for a booking.
- Auth: Bearer token required.
- Response: latest payment state for the booking.

4. POST /payments/refund
- Purpose: Issue a refund for a booking.
- Auth: Bearer token required.
- Request fields: bookingId, optional reason.
- Response: updated payment status = refunded.

5. POST /notifications/send
- Purpose: Internal service-to-service notification creation.
- Auth: Internal key required via x-internal-service-key.
- Request fields: userId, bookingId, paymentId, channel, destination, message.
- Response: created notification record.

6. GET /notifications/history
- Purpose: Retrieve notification history for a user.
- Auth: Bearer token required.
- Query: userId (optional when derived from authenticated identity).
- Response: array of notification records.

The full API contract is documented in swagger.yaml.


9.4 Inter-Service Communication

This microservice integrates with at least one other microservice as required.

A. Outbound integration
- Payment Service calls Auth Service endpoint GET /auth/verify to validate bearer tokens.
- This occurs in authentication middleware before protected handlers run.

B. Inbound integration
- Booking Service calls POST /payments after booking confirmation.
- Booking/Frontend can call GET /payments/:bookingId to fetch status.

C. Internal communication pattern
- After successful payment creation, the payment logic triggers notification creation logic.
- Internal notification endpoint is guarded by internal service key to prevent unauthorized external use.

Validated integration evidence:

9.6 Security Measures

Application-level security controls:
1. Bearer token authentication for protected endpoints.
2. Internal service key authorization for internal notification route.
3. Input validation with express-validator on request payloads and parameters.
4. Security headers via Helmet middleware.
5. CORS configuration for service communication.
6. Secret/config management via environment variables (.env ignored in git).

Data protection and least privilege concepts:
- Sensitive credentials are not hardcoded in source code.
- Internal operations are separated from public routes.
- Only required fields are accepted through strict validation rules.

DevSecOps controls:
- SonarCloud configuration included and wired into CI flow.
- Linting and test gates reduce the chance of vulnerable/unreliable code reaching deployment.

Cloud security implementation approach:
- Managed container platform deployment with secret references.
- HTTPS ingress enabled in deployment manifest.
- Additional cloud-side controls (IAM role assignment, network restrictions) are configured at deployment stage and must be shown during viva.
